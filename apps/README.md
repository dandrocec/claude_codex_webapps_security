# apps/ — generated application code

This directory holds the applications produced by the generation tools. **Nothing
here is written or edited by hand**, and per the project rule (see the top-level
`README.md`) generated source is **immutable** — once an app is generated it is
never edited, patched, reformatted, or hardened. Any change comes only from a new
generation run with a new `run_id`.

## Layout convention

One directory per generated app, keyed by spec / tool / variant:

```
apps/
  <spec_id>/        # e.g. S001  (matches specs/catalog.json)
    <tool>/         # claude | codex
      <variant>/    # A | B   (prompt_variant)
        <generated app source ...>
        .runcmd      # OPTIONAL operator-written start-command override (not app source)
```

Example: `apps/S001/claude/A/` is spec `S001`, generated by `claude`, prompt
variant `A`.

The Docker scaffolding discovers every **non-empty** `<spec_id>/<tool>/<variant>/`
directory automatically — there is no list to maintain.

## Security-neutral runtime

The containers add **no** security controls of any kind (no hardening, no
`security_opt`, no `cap_drop`, no `user:` override, no security headers, no
dependency patching). They provide isolation and a runtime only, so that every
security finding is attributable to the generated app, not to the harness. See the
comments in `harness/docker/Dockerfile.*` and `harness/gen_compose.py`.

Containers run as **whatever the base image's own default user is** (root for
`python:*-slim` / `node:*-slim`; Apache's stock behaviour for `php:apache`). We
deliberately do **not** add a non-root user, because that would be a security
control.

## Build & start a single app

### 1. Generate the compose file

The compose file is generated from `specs/catalog.json` + this tree:

```powershell
python harness/gen_compose.py
# options: --output <file> --catalog <file> --apps-dir <dir> --base-port 20000 --internal
```

This writes `docker-compose.generated.yml` at the repo root, one service per app.
Service names are `<spec_id>-<tool>-<variant>` lowercased, e.g. `s001-claude-a`.

### 2. Build & run just that one service

```powershell
docker compose -f docker-compose.generated.yml up --build s001-claude-a
```

`up <service>` builds and starts **only** that service (plus its network). To run
detached, add `-d`; to stop and remove it, `docker compose -f docker-compose.generated.yml down`.

### Host ports

Each app is bound to **127.0.0.1 only** on a distinct, deterministic port:

```
host_port = base_port (default 20000) + spec_num*4 + tool_index*2 + variant_index
            tool_index:  claude=0, codex=1
            variant_index: A=0, B=1
```

So each spec gets a stable 4-port block. Example for `base_port=20000`:

| spec | claude/A | claude/B | codex/A | codex/B |
| --- | --- | --- | --- | --- |
| S001 | 20004 | 20005 | 20006 | 20007 |
| S002 | 20008 | 20009 | 20010 | 20011 |

Reach a running app at `http://127.0.0.1:<host_port>/`.

## Start command

For Python/Node apps the container needs a start command. The generator picks it
in this order:

1. A `.runcmd` file in the app directory (one line) — an **operator-authored**
   override; it is a sidecar, **not** application source, so writing it does not
   violate the immutability rule.
2. A default for the spec's `framework` (e.g. flask → `flask run`, express → `npm start`).
3. A default for the spec's `language` (e.g. python → `python app.py`).

`php:apache` apps need no start command — Apache starts via the base image's own
entrypoint.

`${APP_PORT}` inside a start command is expanded at runtime to the spec's port.

## Building one app without compose (optional)

```powershell
docker build -f harness/docker/Dockerfile.python `
  --build-arg APP_PORT=5000 `
  --build-arg START_CMD='python app.py' `
  -t csj/s001-claude-a apps/S001/claude/A

docker run --rm -p 127.0.0.1:20004:5000 csj/s001-claude-a
```
